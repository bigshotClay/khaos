A call site that passes a buffer handle where an image handle is expected compiles and links without error, manifesting only as a Vulkan validation layer error at runtime or, worse, a GPU fault. # Story Context

## Source

GitHub issue #6 — VK-2: Typed handle layer — @JvmInline value classes for all Vulkan handles

## Design Tensions

### PipelineHandle.reusable
A standard `@JvmInline value class` on JVM permits exactly one backing property. `PipelineHandle` requires both a `Long` (the Vulkan handle) and a `Boolean` (`reusable`). Options:

1. **Bit-pack** — store `reusable` in the high bit of the `Long`. Zero overhead, but obscures intent and requires masking on every access.
2. **Regular data class** — drop `@JvmInline` for `PipelineHandle` only. Loses allocation-free guarantee for that one type; all other handles remain inline.
3. **Separate types** — `ReusablePipelineHandle` and `SingleUsePipelineHandle`, each a single-field inline class. Exhaustive but doubles the pipeline handle surface.

Decision: Option 2 (regular data class). Bit-packing (Option 1) was rejected because encoding the reusable flag in the high bit of a native handle value risks corruption when the raw Long passes through JNI to the Vulkan driver. Separate types (Option 3) were rejected to avoid doubling the pipeline handle API surface for VK-3 through VK-8. PipelineHandle carries one heap allocation per instance that all other handle types avoid — this cost is documented via KDoc on the class. AC5 requires this explanation be present in the implementation.

### Zero-allocation test strategy
The issue asks for a "size check via reflection or bytecode inspection." In practice:
- `KClass.isValue == true` confirms the compiler treated the type as a value class.
- Full allocation absence requires JMH or a Java agent (e.g., JAMM) — out of scope for a unit test.
- A reasonable test asserts `isValue == true` and spot-checks that the class has no fields beyond the single backing property via reflection.
