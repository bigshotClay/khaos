## Ruling

The implementation satisfies all six acceptance criteria.

**AC1 (Sealed exhaustiveness):** The `VulkanOutcome<out T>` sealed class with four variants (`Success`, `Degraded`, `RecoverableFailure`, `UnrecoverableFailure`) enforces Kotlin `when`-expression exhaustiveness at compile time. A `when` expression covering all four variants compiles without `else`; one omitting any variant fails to compile. AC1 is satisfied. CASE-2026-002 established this same mechanism as compliant; the v3 implementation is structurally identical.

**AC2 (Diagnostic message inspection):** The diagnostic test in AC2-P1 asserts both `exitCode == COMPILATION_ERROR` and `messages.lowercase().contains("exhaustive")`. Cyclops's Defense 1 is correct: Kotlin's sealed-exhaustiveness error (`'when' expression must be exhaustive`) is the only common diagnostic that uses the word "exhaustive" as part of its message. Batman's Charge 1 postulated a classpath error containing "exhaustive" — this is speculative. The `inheritClassPath = true` setup ensures all necessary types are resolved; classpath failures do not produce "exhaustive" in their messages. Both assertions together satisfy AC2's requirement to distinguish sealed-exhaustiveness failures from unrelated errors. AC2 is satisfied.

**AC3 (Smart-cast yields statically-typed T):** The positive tests in AC3-P1 and AC3-P2 explicitly declare `val resource: String = outcome.resource` and `val resource: Int = outcome.resource` after `is VulkanOutcome.Success` smart-casts. These declarations would fail to compile if `.resource` resolved to `Any?` — they do not, because the `out T` covariant hierarchy allows Kotlin to smart-cast to `VulkanOutcome.Success<String>` (or `<Int>`). The negative test in AC3-N1 demonstrates that a star-projected `VulkanOutcome.Success<*>` yields `resource: Any?`, causing a `String` return type mismatch. Cyclops's Defense 2 is correct: Kotlin treats unchecked casts as warnings, not errors; the COMPILATION_ERROR in AC3-N1 comes exclusively from the `Any?` ≠ `String` type mismatch. AC3 is satisfied.

**AC4, AC5, AC6 (Variant property access):** `Degraded.indicator`, `RecoverableFailure.cause`, and `UnrecoverableFailure.cause` are typed properties returning `SwapchainStatus` and `VulkanError` respectively. Smart-casts after `is`-checks expose these properties at the correct static type. `UnrecoverableFailure` is structurally distinct from `RecoverableFailure` (`shouldNotBeInstanceOf` passes). CASE-2026-002 (Deadpool, PASS) established compliance for analogous ACs in v2. AC4, AC5, and AC6 are satisfied.

**Both charges are DISMISSED.** Batman's Charge 1 relies on a speculative classpath scenario that the test setup structurally prevents. Batman's Charge 2 misattributes the compile error source: Kotlin's error model assigns warnings to unchecked casts and errors to type mismatches. Neither charge meets the evidentiary standard of an implementation violation.

## Critic Review

*Iron Man — Post-Verdict Critic Review*

The courtroom ran cleanly, but both sides left engineering value on the table.

**On the prosecution:** Batman's Charge 1 — that `shouldContain "exhaustive"` could match a non-exhaustiveness error — was speculative from the jump. The charge deserved a concrete counter-example or it shouldn't have been filed. Raising hypothetical classpath error messages without reproducing one is the advocacy equivalent of filing a patent on an idea you haven't built. Future prosecutors: if you can't trigger the failure mode in a test, it's not a charge — it's a footnote.

Batman's Charge 2 was technically sharper. The observation that `as VulkanOutcome.Success<*>` generates an unchecked cast warning, not an error, is accurate. But the charge misfired because it conflated the warning source with the error source. The compile error in AC3-N1 comes from the return type mismatch (`Any?` ≠ `String`), not from the cast itself. Distinguishing error cause from error trigger is basic type theory — prosecutorial due diligence should have parsed that before filing.

**On the defense:** Cyclops performed well. Defense 1 (classpath errors don't produce exhaustive in their messages) was precise and dispositive. Defense 2 correctly isolated Kotlin's error model — unchecked casts are warnings; type mismatches are errors. Both defenses were grounded in the actual compiler behavior, not extrapolation.

**What the courtroom missed:** Neither side examined the compile-testing architecture as a whole. Seven separate `KotlinCompilation` instances with `inheritClassPath = true` works, but it's also the slowest possible test design for this coverage surface. A future advocate could profitably charge that the test suite sacrifices build-time budget for test isolation that doesn't require separate compilations. That charge wouldn't threaten any AC — it's a quality-of-implementation concern — but it's exactly the kind of signal that separates a passing implementation from a shippable one.

**Takeaway for future advocates:** When testing compile-time properties, know whether your assertion is testing the compiler's error *message* or its error *exit code*. This case exposed both — AC2 tested the message, AC1 tested the exit code. Charges that conflate the two will be dismissed quickly.

The verdict is correct. The implementation is sound.
