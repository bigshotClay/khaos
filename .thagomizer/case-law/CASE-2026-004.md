## Ruling

AC3 does not prohibit the backing `val handle: Long` property on inline value class handles — JNI bridge layers must expose the raw Long for native calls, making this a JVM representation detail, not an API position. The `get*` filter exclusion in AC3 tests is correct and documented. AC4 "companion factory" is satisfied by the companion NULL sentinel pattern — the "so that" clause defines purpose (preventing bypass), not architecture (callable method). All five critical ACs satisfied in iteration 2.

## Critic Review

*Wolverine — Post-Verdict Critic Review*

Clean run. Both charges were technically honest observations that couldn't close the gap to actual violations. Here's what the advocates should know for next time.

**On the prosecution:** Charge 1 (AC3 filter) was the stronger play. The observation is correct — `getHandle()` IS excluded and DOES return `long`. But the charge died because it never answered the obvious follow-up: if that's a violation, how do you write a JNI bridge handle layer? The prosecution had to either propose an alternative design or concede the point. It did neither. Future prosecutors: if your charge leads to an impossible implementation, your charge is wrong. Diagnose that before filing.

Charge 2 (companion factory) imported a technical definition ("callable mechanism for constructing instances from arbitrary inputs") that doesn't exist in the spec. That's speculation dressed as statutory interpretation. The spec says "companion factory so that the null sentinel cannot be bypassed" — the purpose clause was the whole game, and the prosecution tried to rewrite it as an architecture clause. Cyclops's defense caught this immediately.

**On the defense:** Both defenses were tight. Defense 1's "handle POSITION vs backing property" distinction was exactly right and supported by the spec background. Defense 2's "so that" clause analysis was the strongest argument in the room — read the purpose, not the pattern. The only gap was not citing Hank Pym's own framing: even the reviewer who flagged the factory concern called it "VK-3 blocked," not "VK-2 violated." That would have ended Charge 2 in the first paragraph.

**What the courtroom missed:** Nobody examined the iteration 2 fix holistically. The filter chain in HandleSpec.kt (iteration 2) is correct and documented in the manifest, but the manifest justification ("not API positions") begs the question of what "API position" means for a JNI bridge type. A future prosecutor with stronger spec-reading could challenge this framing in a similar story. The ruling in CASE-2026-004 now establishes that the JNI backing property is not an "API position" for AC3 purposes — that precedent is worth knowing.

**Takeaway:** In JVM Vulkan bridge stories, the tension between "no raw Long in API" and "must expose Long for JNI" is a recurring design constraint. Future test suites should document the exclusion rationale inline in the test, not only in the manifest, so the trade-off is visible at the call site.
