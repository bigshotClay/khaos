## Session — SPIKE-SHADER-1 + SPIKE-SHADER-2 (Issues #1 and #2)

**What happened:** Produced decision documents for both shader pipeline spikes. Both spike hypotheses were invalidated — different tools from the ones initially proposed are the right ones.

**Design decisions:**

- **Shader compilation: glslc subprocess, not LWJGL shaderc JNI** — LWJGL native extraction is problematic in Gradle daemon JVM; no ecosystem precedent for build-time use; spirv-val has no JVM bindings at all. Subprocess via `@CacheableTask` is the universal ecosystem pattern.
- **Shader binding codegen: standalone Gradle task, not KSP** — KSP cannot declare external file dependencies to Gradle; incremental builds break silently on JSON input changes. Custom `@CacheableTask` with `@InputFiles`/`@OutputDirectory` is correct for external-file-primary codegen.
- **SPIR-V reflection tool: spirv-cross `--reflect`** — already in Vulkan SDK alongside glslc; no extra dependency. `spirv-reflect-kt` noted as viable pure-JVM migration path.
- **Three-stage task chain settled:** `compileShaders` (glslc) → `reflectShaders` (spirv-cross) → `generateShaderBindings` (Kotlin codegen) → `compileKotlin`

**Observations:**
- Both spikes named a hypothesis tool (LWJGL shaderc, KSP) that turned out wrong for the use case. Invalidating clearly was the value, not finding workarounds.
